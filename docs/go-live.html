security group

Public internet reaches Traefik, not every container.

inbound-rules.txt
22/tcp    SSH        my IP only
80/tcp    HTTP       public
443/tcp   HTTPS      public
3001      Next.js    closed
5432      Postgres   closed
3000      Grafana    closed directly
9090      Prometheus closed
3100      Loki       closed

Common real mistake

5432/tcp Postgres public to 0.0.0.0/0

That makes the database reachable from the internet. Even with a password, this is unnecessary risk.

Every public port is an attack surface.07 / 26

dockerfile

Separate build time from runtime.

Beginner pattern

Large floating base image, weak cache, may copy local files and secrets, keeps build tools in runtime.

Dockerfile
# common beginner Dockerfile
FROM node
WORKDIR /app
COPY . .
RUN npm install
RUN npm run build
CMD ["npm", "start"]

# better pattern
FROM node:24-slim AS builder
WORKDIR /app
COPY package.json bun.lock ./
RUN bun install --frozen-lockfile
COPY . .
RUN bun run build

FROM node:24-slim AS runner
ENV NODE_ENV=production
COPY --from=builder /app/apps/web/.next/standalone ./
CMD ["node", "apps/web/server.js"]

compose

Dockerfile builds one image. Compose runs the system.

Compose describes services, networks, volumes, restart policy, env files, and health dependencies.

docker-compose.prod.yml
services:
  web:
    image: ghcr.io/org/demo-harness:${APP_VERSION}
    restart: unless-stopped
    env_file: .env.production
    networks: [internal, proxy]
    depends_on:
      postgres:
        condition: service_healthy

  postgres:
    image: postgres:17
    restart: unless-stopped
    networks: [internal]
    volumes:
      - postgres_data:/var/lib/postgresql/data

stateful data

App containers are replaceable. Database data is not.

state-model.txt
web container
  replace on every deploy

postgres container
  can be recreated

postgres_data volume
  must persist
  must be backed up

Production upgrade

Container Postgres is useful for learning. For a serious product, consider managed Postgres such as Amazon RDS.

pg_dump is not enough unless restore is tested.

Backup is only real when restore works.11 / 26

pipeline

CI checks code. CD moves the verified artifact to EC2 and fails if health verification fails.

registry

Never rely on latest for rollback.

The SHA tag is the source of truth. Human-friendly tags are shortcuts.

deploy.sh
ghcr.io/org/demo-harness:a1b2c3d
ghcr.io/org/demo-harness:main

APP_IMAGE=ghcr.io/org/demo-harness:a1b2c3d \
docker compose -f docker-compose.prod.yml up -d

curl -fsS https://app.example.com/healthz

demo-harness:main-20260630-1015
demo-harness:prod-current
demo-harness:prod-previous

runtime config

Secrets should not be inside the Docker image.

The image contains code. The server or secret store injects runtime configuration.

EC2 .envGitHub secretsSSMSecrets Manager
.env.production
APP_IMAGE=ghcr.io/org/demo-harness:a1b2c3d
POSTGRES_PASSWORD=long-random-password
DATABASE_URL=postgresql://postgres:${POSTGRES_PASSWORD}@postgres:5432/demo-harness
CORS_ORIGIN=https://app.example.com
GRAFANA_ADMIN_PASSWORD=long-random-password

Bad:
  COPY .env .env
  bake DATABASE_URL into image

Better:
  server/runtime injects DATABASE_URL

release safety

Deploy is only half the story. We also need a way back.

Rollback is easy only when database changes are compatible.

release-record.txt
release_id: 2026-06-30-001
commit: a1b2c3d
image: ghcr.io/org/demo-harness:a1b2c3d
deployed_by: github-actions
healthz: passed
previous_image: ghcr.io/org/demo-harness:9f8e7d6

APP_IMAGE=ghcr.io/org/demo-harness:9f8e7d6 \
docker compose -f docker-compose.prod.yml up -d

Safer migration:
1. add nullable column
2. deploy app that reads old and new shape
3. backfill data
4. make stricter constraint later

tradeoffs

VPS gives control, but also responsibility.

Class demo

One EC2 instance with Compose is transparent and teachable.

  • EC2 instance
  • EBS disk
  • Elastic IP
  • domain name
  • backup and monitoring storage

Paid product upgrade

Consider stronger managed pieces when customers depend on it.

  • RDS for durability
  • managed backups
  • private subnets
  • IAM/SSM instead of direct SSH
  • alert routing
The point is not VPS over PaaS. The point is understanding the tradeoff.23 / 26
