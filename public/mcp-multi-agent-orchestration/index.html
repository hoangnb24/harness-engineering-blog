Chào mọi người. Hôm nay mình nói về MCP và multi-agent orchestration dưới góc nhìn rất thực tế cho web application development. Mình không muốn biến buổi này thành danh sách tool đang hot. Why this topic?

From tool demo to development architecture

The interesting question is no longer “which MCP server should I install?” It is “how do I organize context so development gets faster without losing control?”

student-workflow.txt
# old loop copy log -> paste into chat -> hope the model understands # stronger loop issue + repo + browser + docs + checks -> scoped agent work -> verified patch
Khi bắt đầu dùng AI coding assistant, trải nghiệm ban đầu thường là chat với model và yêu cầu nó viết code. Cách đó có giá trị, nhưng chưa đủ cho workflow phát triển phần mềm nghiêm túc. Ecosystem đã đi xa hơn: GitHub, Figma, Vercel, Supabase, Stripe, Playwright và nhiều nền tảng khác đều đưa context vào agent. Câu chuyện không còn là cài server nào, mà là tổ chức context như thế nào để development nhanh hơn, ít lỗi hơn, và vẫn kiểm soát được rủi ro.
The thesis

Access, attention, and the real game

MCP is the access protocol.It gives agents a standard way to reach files, repos, browsers, databases, logs, design, docs, and product systems.
Multi-agent orchestration is the attention topology.It decides which worker focuses on code, docs, browser, security, or synthesis.
Context management is the game.Better outcomes come from the right evidence, scoped to the right agent, then verified.
Access != workflowAttention != more tokens
Nếu chỉ nhớ một slide, mình muốn mọi người nhớ slide này. MCP là access protocol. Nó cho agent cách chuẩn để chạm vào hệ thống bên ngoài. Multi-agent orchestration là attention topology: agent nào tập trung vào code, docs, browser, security, hay synthesis. Context management mới là game thật sự. Model mạnh hơn và context window lớn hơn có ích, nhưng nếu đưa sai context, quá nhiều context, hoặc context chưa verify, agent vẫn đi sai rất nhanh.

The Context Management Game

Five moves before the code is trustworthy

01

Acquire

Find the right information from code, docs, design, runtime, database, and product systems.

02

Scope

Send the smallest useful subset to the right agent.

03

Isolate

Move noisy exploration, logs, screenshots, and test output into bounded worker contexts.

04

Compress

Return distilled evidence, decisions, references, and next actions.

05

Govern

Control read/write permissions, approvals, logging, and sensitive data boundaries.

Small context, high evidenceNot a prompt trick
Mình chia context management thành năm động tác. Acquire là lấy đúng thông tin từ code, docs, design, runtime, database, product system. Scope là không nhét hết vào prompt, mà gửi phần nhỏ nhất đủ dùng cho đúng agent. Isolate là để việc nhiều noise như log dài, screenshot, test output vào worker context riêng. Compress là trả về evidence đã chưng cất. Govern là kiểm soát quyền đọc, quyền sửa, approval, logging, và dữ liệu nhạy cảm.
Context pollution
More context can become more distraction.
bad-thread.log
stack trace x 400 lines test retries x 9 three unrelated files old decision from yesterday important clue hidden in the middle
Một phản xạ tự nhiên là: context window lớn hơn thì cứ đưa hết mọi thứ vào. Nhưng context lớn hơn không đảm bảo model tập trung tốt hơn. Nếu thread chính có quá nhiều log, stack trace, test output, file đọc thử, thông tin quan trọng có thể bị chìm. Đây là context pollution. Mục tiêu không phải là nhồi nhiều nhất có thể, mà là thiết kế luồng context: cái gì cần đọc, ai đọc, đọc trong bao lâu, và trả về dạng nào.
MCP in one diagram

HOST -> CLIENT -> SERVER

Host

LLM app or IDE: Codex, Cursor, ChatGPT, VS Code. This is where the agent experience lives.

Client

Connection layer inside the host: JSON-RPC, auth, and consent.

Server

External context and action surface: resources, prompts, and tools.

MCP does not decide the workflowIt standardizes access
MCP có thể hiểu đơn giản là chuẩn kết nối giữa AI application và hệ thống bên ngoài. Host là ứng dụng AI hoặc IDE: Codex, VS Code, Cursor, ChatGPT. Bên trong host có client, client nói chuyện với MCP server. Server expose resources, prompts, tools. Điểm quan trọng: MCP không tự quyết định workflow. MCP chỉ chuẩn hóa cánh cửa truy cập. Agent nào được đi qua cánh cửa nào, lúc nào, với quyền gì là bài toán orchestration.
From plug to protocol surface

The direction is infrastructure, not novelty

The recent direction frames MCP as moving toward auth, consent, schema, tasks, cacheability, tracing, UI, and lifecycle.

A

2025-11-25 stable direction

Authorization discovery, incremental consent, elicitation, sampling tools, JSON Schema 2020-12.

B

2026-07-28 release-candidate direction

Stateless core, MCP Apps, Tasks extension, authorization hardening, cacheable lists, trace context.

C

Production implication

MCP becomes a surface for auth, observability, UI, and lifecycle.

Điểm thú vị là MCP đang đi từ tool plug thành protocol surface nghiêm túc hơn. Các thay đổi gần đây không chỉ là thêm tool mới. Nó đi vào authorization, consent, schema, task lifecycle, cacheability, tracing, và MCP Apps. Với bài talk này, mình không cần mọi người nhớ hết từng thay đổi. Chỉ cần nhớ hướng đi: MCP đang trở thành lớp hạ tầng để agent làm việc với thế giới bên ngoài.

The good and the ugly

MCP raises capability and risk together

Better context creates better agents. Broader tool access creates broader failure modes.

The Good

Less guessing

Agents can ground decisions in repo, docs, browser, database, and runtime context.

The Good

Faster dev loops

Context acquisition and verification move closer to the code change.

The Ugly

Tool overload

Every tool definition costs context and makes tool choice harder.

The Ugly

Scope mistakes

Prompt injection, broad writes, cost explosion, and parallel conflicts become architecture concerns.

MCP có mặt rất tốt: giảm guessing, agent grounded hơn, dev loop nhanh hơn, audit workflow tốt hơn nếu trace tool call rõ. Nhưng mặt xấu cũng thật: tool overload, prompt injection, scope mistake, cost explosion, parallel write conflict. Với multi-agent, nhiều agent nghĩa là nhiều context, nhiều tool call, nhiều token, và nếu cùng sửa code thì conflict tăng nhanh.
Server selection filter

Do not add a server just because it exists

1

Does it improve context quality?

Context7 gives current docs instead of model memory.

2

Does it reduce human copy-paste?

GitHub, Figma, and Supabase bring structured project context directly.

3

Does it close a verification loop?

Playwright turns browser behavior into evidence.

4

Can it be scoped safely?

Read-only, project-scoped, allowlisted, and approval-gated writes.

Chọn MCP server bằng bốn câu hỏi. Server này có cải thiện chất lượng context không? Có giảm copy-paste thủ công không? Có đóng verification loop không? Và có scope an toàn được không? Docs MCP read-only thường đủ. GitHub có thể đọc issue trước, comment hoặc merge phải approval. Database nên development branch và read-only khi inspect schema. Billing hoặc auth phải giới hạn scope và approval rõ.

Reduce design-to-code guessing

Use Figma when a real design exists. Use shadcn context when the live demo needs reliable component vocabulary.

shadcn MCP

If there is no prepared Figma frame, keep Figma conceptual and demo repo, test, browser, and Codex agents.

Nhưng nếu demo không có frame chuẩn bị sẵn, đừng demo Figma. Trong workshop, Figma là ví dụ concept; live demo dựa trên repo, test, browser, Codex agents. shadcn MCP vẫn rất tốt vì giúp agent ở trong vocabulary của design system.

Database context

Useful, powerful, and dangerous

Supabase and Neon can turn schema assumptions into facts, but database access is an operations surface.

Useful context

Schema and relationships

Migrations, SQL behavior, logs, advisors, and generated types.

Safety rule

Development branch first

Read-only by default, project scope, feature restriction, approval for writes.

Do not guess

Inspect before typing

Use schema and constraints instead of inventing table fields or relationships.

Production data

Not all useful context belongs in the model

Separate inspection, privacy, and write authority.

Database context rất mạnh vì giúp agent ngừng đoán. Thay vì đoán table có field gì, agent inspect schema. Thay vì đoán relationship, nó đọc constraint. Nhưng đây cũng là vùng nguy hiểm vì database MCP có thể là operations surface. Default của mình là development branch trước, read-only trước, project scope rõ, feature restriction rõ, và mọi write hoặc destructive action phải có approval.

Gateway / broker pattern

Do not expose every server to every agent

1

Use a gateway or broker

Decide which toolsets are visible to each agent.

2

Reduce schema load

Tool allowlists and deferred loading keep the context window cleaner.

3

Map scopes to roles

OAuth scopes and read/write policies should match responsibilities.

4

Trace tool calls

Observability is how teams debug agent behavior and audit risk.

Khi team lớn hơn, pattern hợp lý không phải mỗi agent tự kết nối trực tiếp đến mọi MCP server. Pattern hợp lý hơn là gateway hoặc broker. Docker MCP Gateway quản lý profile, routing, credentials, lifecycle, container isolation, và tracing cho nhiều server. Gateway quyết định docs_researcher thấy docs tool, browser_debugger thấy browser tool, db_mapper thấy database read-only tool. Lợi ích là giảm schema load, tăng governance, và có trace khi workflow fail.

Nếu dùng multi-agent mà không có contract, mình chỉ tạo ra hỗn loạn song song. Contract nói rõ agent này được suy nghĩ về phần nào, dùng tool nào, sửa gì, và trả evidence dạng nào. Purpose phải cụ thể. Input context phải có starting point. Allowed tools phải hẹp theo role. Write scope chống conflict. Approval rules phải cụ thể. Output schema ép worker trả về decision-grade summary, không phải toàn bộ log.
Codex web-dev cycle

From intake to observed ship

1

Context intake

Main agent reads request, AGENTS.md, repo docs; spawns read-only explorers when space is wide.

2

Plan and decomposition

Convert findings into a story-sized plan, owners, acceptance criteria, and write boundaries.

3

Implementation

Main agent integrates; subagents handle bounded UI, API, DB, or copy work when ownership is clean.

4

Validation

Run tests, browser checks, security review, and summarize high-noise logs.

5

Ship and observe

Prepare PR summary; use GitHub, Vercel, and Sentry only as needed, with production read-only.

Áp dụng vào Codex, cycle bắt đầu bằng context intake: main agent đọc request, AGENTS.md, repo docs, và spawn explorer read-only nếu rộng. Sau đó planning: biến context đã nén thành plan nhỏ, acceptance criteria, write boundary. Implementation chỉ parallel khi boundary sạch. Validation là tests, browser, security review, log triage. Cuối cùng ship and observe: PR summary, GitHub/Vercel/Sentry khi cần, production context read-only và scope chặt.

Mình không muốn kết thúc bằng hype. MCP và multi-agent tăng productivity, nhưng cũng tăng capability và risk. Tool output có thể chứa prompt injection. OAuth scope quá rộng làm agent thấy hoặc làm nhiều hơn cần thiết. Production data đi vào context tạo privacy và compliance risk. Mitigation thực tế: official server first, read-only mặc định, tool filtering, approval-gated writes, trace mọi tool call quan trọng. Context is power; orchestration giữ power đó legible.
