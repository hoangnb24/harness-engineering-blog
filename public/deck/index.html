The problem

Most repos are built for humans reading familiar code.

What the agent sees

  • A chat prompt
  • A shallow snapshot of files
  • Maybe a README

What goes wrong

  • Edits code before reading product intent
  • Constraints live only in chat history
  • Validation expectations discovered too late
  • Architecture tradeoffs repeated, not inherited
  • Big requests don't get broken into reviewable stories

"Early progress was slower than we expected, not because Codex was incapable, but because the environment was underspecified." — OpenAI

The failure modes are well known to anyone who's used an agent: it edits before it reads the spec, it repeats the same architectural decisions over and over, the validation step is "I think it works." OpenAI themselves said progress was slow not because the model was weak, but because the *environment* was underspecified. That's the gap our repo fills.

What repository-harness is

A repository-level operating harness you install into any project.

It gives coding agents the missing project context they need before they change code:

  • Where to start — AGENTS.md is the stable shim every session reads
  • What the product contract says — docs/product/ is the source of truth
  • How risky the work is — tiny / normal / high-risk lane classification
  • What proof is required — the test matrix and story verification commands
  • Which decisions to inherit — durable docs/decisions/ log

The app is what users touch. The harness is what agents touch. — README.md

Five things every agent needs to answer. Without a harness, it guesses. With one, it knows. The README's tagline is the most-quoted line: "The app is what users touch. The harness is what agents touch." That's the mental model.

Install

One curl command drops the harness into any project.

macOS / Linux

curl -fsSL "https://raw.githubusercontent.com/hoangnb24/repository-harness/main/scripts/install-harness.sh" \
  | bash -s -- --yes

Windows PowerShell

& ([scriptblock]::Create((irm "https://raw.githubusercontent.com/hoangnb24/repository-harness/main/scripts/install-harness.ps1"))) -Yes

Installs AGENTS.md, docs/, scripts/, and the Rust harness-cli binary.

This is the "anyone can run it now" moment. The installer is non-destructive by default — it stops if AGENTS.md already exists unless you pass --merge or --override. For Claude Code users, pass --claude so the harness loads via CLAUDE.md. We'll come back to this in the live demo.

  .github/

+ an optional CLAUDE.md shim if you use Claude Code.

This is the file map. The four things that matter most: AGENTS.md is the agent's first read. docs/ is the operating policy. harness-cli is the durable-state tool. The schema is version-controlled so the database can be regenerated. No app code is scaffolded — that's intentional, we don't want to lock a stack on someone.

Risk lane classification

Every request enters the intake gate before any code changes.

tiny

low-risk docs, copy, narrow edits. Skips story-packet overhead.

normal

story-sized behavior, bounded blast radius. Story packet required.

high-risk

touches auth, data, contracts, multiple platforms. Human gate + decision record.

Hard gates: auth, authorization, data loss, external systems, removing validation.

Defined in docs/FEATURE_INTAKE.md

This is the "humans steer" part made concrete. The agent doesn't pick lanes by vibes — there's a checklist of risk flags and hard gates. 0–1 flags = tiny/normal. 4+ flags = high-risk. Any hard gate = high-risk unless a human explicitly narrows scope. This is one of our biggest deviations from OpenAI's framing — they don't publish a comparable gate.

The task loop

human intent / prompt
    |
    v
intake   ── > classify input type, choose lane, record with harness-cli intake
    |
    v
story    ── > create or update story packet, link product docs
    |
    v
trace    ── > record what happened, score-trace, friction captured
    |
    v
backlog  ── > friction + interventions + audit → propose improvements
    |
    v
growth   ── > harness evolves from friction. the repo compounds.

Every loop run leaves a durable record in harness.db.

This is the orchestration. Every task goes through this loop. The beautiful part: every loop run leaves a *durable record*. The agent's work is auditable after the fact. The harness can be improved *from evidence*, not from vibes. That's H3 active observability, and it's the seed of H5 self-improvement.

The agent wrote code first, asked questions never.

This is what an agent does in a typical repo. It makes assumptions about auth, schema, response shape, test coverage. It opens a PR. A human reviewer (or a stricter agent reviewer) finds the gaps. Each gap becomes a follow-up. This is the 1.5-PR-per-day world. It's the minimum-viable agent loop.

The agent read the policy first, then wrote code.

Same prompt, same agent. But now there's a policy. The agent is told: classify the work, link the story, match the existing envelope, follow the auth decision, prove it. Notice friction=0. The historical frame

From an arXiv survey: Externalization in LLM Agents (arXiv 2604.08224)

1960s–2010s      weights       →  capability lives inside the model
2018–2023        context      →  capability moves into the prompt + retrieval
2024–2026        harness      →  capability moves into the environment around the model

Three forms of externalization: memory (state), skills (procedures), protocols (interaction), unified by the harness.

Source: arxiv.org/abs/2604.08224

This is the academic frame for why harnesses matter. The arXiv 2604.08224 survey argues that agent progress comes less from changing model weights and more from externalizing cognitive burden into the environment. Memory externalizes state. Skills externalize procedures. Protocols externalize interaction. The harness is the unification layer. Repository-harness is the most ambitious form of this: the *whole repository* is the externalization surface.

Harness compliance went from 80.6% to 100% across five phases. Same agent. Same model. Same prompts. Different environment.

Source: harness-benchmark/benchmark/runs/ (all reports)

This is the proof. The benchmark runs the same 6 tasks against each phase of the harness. Same agent. Same model. Same prompts. The only variable is the harness. Harness compliance — did the agent follow the policy? — went from 80.6% to 100%. Trace quality went from 1.8 to 2.6. Lane accuracy went from 5/6 to 6/6 — meaning the agent always picks the correct risk classification. Functional stayed at 100% because we never changed the agent's coding ability, only its environment. This is the answer to "is this all philosophy?"

Phase 5: the evolution loop

The first step toward H5 self-improving harnesses.

score-context

Measures which docs the agent actually read for a trace. "Did the agent read the right policy before acting?"

audit

Detects durable-state drift. Computes an entropy score across the harness.db. "Is the harness in a healthy state?"

propose

Generates structured improvement proposals from friction + interventions + audit results. "What should we change about the harness?"

$ harness-cli audit           # entropy score, drift categories
$ harness-cli propose         # reviewable improvement proposals
$ harness-cli propose --commit # creates backlog items, never edits policy

Spec: docs/IMPROVEMENT_PROTOCOL.md

Three new commands, all shipped in Phase 5. The key insight: propose does NOT edit the policy. It creates reviewable backlog items. The high-risk changes (architecture, source-of-truth, validation rules) pause for human confirmation. This is the seed of H5. The next research question is: do repeated runs of the proposal loop actually *improve* the harness? That's the closed-loop experiment.

Roadmap

What comes after H4 achieved.

▸ H3 closure — component-level regression attribution in the benchmark. Which of the 11 responsibilities moved?

▸ H5 closure — repeated benchmark outcomes proving the proposal loop. Closed-loop experiment.

▸ Permission enforcement — turn advisory policy into enforced command allowlists. The biggest gap.

▸ Tool usage analytics — measure which docs the agent actually reads. Feed back into score-context.

▸ Long horizon — harnesses that write themselves. The arXiv 2604.25850 frontier.
